## Auth Flow

### Tipo D: API Key simple

1. Usuario genera API Key en: `https://v0.app/chat/settings/keys`
2. CLI guarda key en config via `api-key-wizard` block (masked prompt + live validation)
3. Cada request: `Authorization: Bearer <V0_API_KEY>` en header
4. Opción alternativa: env var `V0_API_KEY` (sin config file)

No hay OAuth2, no hay browser, no hay cookies. Token no expira — solo revocación manual.

### Rate Limit Preflight (patrón único de v0-cli)

Antes de cualquier llamada costosa, CLI verifica rate limits:

```typescript
// src/lib/rate-limits/preflight.ts
await assertRateLimitOk(client, profile, { outputMode: mode });
// Si remaining < threshold (default: 50) → lanza CliError code:'rate_limit_low'
// Grace period accounts: bypass automático del gate
// --force: override del gate
```

Cache local de rate limits en `~/.config/v0/rate-limits-cache.json`. TTL configurable (`profile.rate_limits.refresh_interval_seconds`, default 60s).

## Trust Ladder T0–T3

v0-cli implementa la taxonomía completa de riesgo. Toda operación tiene nivel asignado en `src/lib/trust/ladder.ts`:

| Nivel | Semántica | Ejemplos | Mecanismo |
|-------|-----------|----------|-----------|
| T0 | Reads / introspección | `auth status`, `chat list`, `rate-limits`, `doctor` | Sin gate — pasa directo |
| T1 | Writes baratos (solo audit) | `chat create`, `msg send`, `project create` | Sin confirmación, audit log |
| T2 | Confirm | `deploy create`, `chat delete`, `env pull` | Prompt `--yes` en TTY; en `--json` lanza error estructurado |
| T3 | Killswitch (intent token) | `deploy delete`, `hook delete`, `mcp-server delete`, `project delete --delete-all-chats` | Requiere `--yes --confirm <token-id>` |

### Intent Tokens (T3)

Single-use tokens HMAC-signed para operaciones destructivas. Flujo:

```bash
v0 intent issue deploy delete --chat abc123    # genera token, devuelve ID
v0 deploy delete --chat abc123 --yes --confirm <token-id>   # consume token
```

Implementación (`src/lib/trust/intent.ts`):
- Token = `v1.<id>.<HMAC-SHA256-sig>`
- Almacenado en `~/.config/v0/intents/<id>.json` con `paramsHash` (SHA-256 del payload)
- Expira por tiempo + se marca `consumedAt` al usarse → imposible reusar
- Secret local generado en `~/.config/v0/intents/.secret` (mode 0o600)

Diferencia vs `--dry-run`: `--dry-run` previene la acción, intent tokens la autorizan con trazabilidad.

## Background Workers

```bash
v0 "landing page hero" --background --json    # fire-and-forget
v0 chat pending --json                        # ver todos los jobs pendientes
v0 chat watch <id>                            # tail live del output
```

Implementación (`src/lib/background/`):
- Registry JSON local: `~/.config/v0/background/<id>.json`
- Log NDJSON por job: `~/.config/v0/background/<id>.log`
- `chat-background.ts` hace tail del NDJSON log con polling 400ms
- `isProcessAlive()` verifica PID del proceso hijo

Útil para agentes: lanzar generación larga, retomar más tarde con `chat wait <id>`.

## Shorthand Routing

v0-cli deduce el subcomando por la forma del argumento:

```
v0 "texto"               → chat create (prompt)
v0 .                     → chat init (cwd)
v0 ./path                → chat init (files)
v0 https://github.com/…  → chat init (repo)
v0 https://v0.app/…      → chat init (template URL)
v0 template_abc          → chat init (template id)
v0 https://…/dist.zip    → chat init (zip)
```

Patrón: `src/index.ts` detecta shape del primer argumento y redirige al command handler.

## Stack

- Runtime: Bun + TypeScript
- HTTP client: `v0-sdk` (npm, wrappa la Platform API)
- Bloques cligentic copiados en `src/cli/`: audit-log, killswitch, doctor, next-steps, api-key-wizard
- Output: `src/lib/output/` (human.ts, json.ts, ndjson.ts) — mismo patrón que cligentic json-mode
- Linter: Biome
- Publicado: `@crafter/v0-cli` en npm

## Gotchas

1. `v0-sdk` es el client oficial de Vercel — wrapper tipado, no HTTP raw. No necesita http.ts custom.
2. `env pull` es T2 aunque es read — expone secrets a disco, por eso nivel elevado.
3. `project delete --delete-all-chats` dinámicamente escala de T2 a T3 (ver `classifyProjectDelete()`).
4. `env delete` escala a T3 si count > 1 (`classifyEnvDelete()`).
5. Background jobs requieren que el proceso padre no muera — usar `nohup` o `detach` si hay shell corta.
6. Schema command (`v0 schema`) devuelve el OpenAPI spec completo — útil para recon de endpoints nuevos.
